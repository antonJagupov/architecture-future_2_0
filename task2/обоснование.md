1. Выделение доменов
На основе бизнес-контекста «Будущего 2.0» и целей независимого развития направлений (клиники, финтех, ИИ, будущие партнёры) предлагается следующее декомпозиция на домены:

|Домен|	Ответственность|	Ключевые сущности|	Потребители внутри компании|
|--|---|----|----|
|Домен «Клиники»|	Ведение медицинской деятельности: пациенты, врачи, назначения, инвентаризация, исследования (без передачи медицинских карт в аналитику)	|Пациент, визит, диагноз, инвентарная единица, врач	|Внутренние ИИ-сервисы (по разрешению), финтех (выставление счетов), BI-пользователи (только обезличенная статистика)|
|Домен «Финтех»|	Финансовые продукты: кредиты, счета, транзакции, банковские операции. Имеет банковскую лицензию	|Счёт, платёж, кредитный договор, клиент-физлицо|	Клиники (оплата услуг), головной офис (отчётность), партнёры (фарма, электроника)|
|Домен «ИИ-сервисы»|	Анализ медицинских данных, построение моделей, выдача рекомендаций врачам	|Модель, инференс, обученные признаки|	Клиники (вспомогательная диагностика), R&D-подразделение|
|Домен «Партнёры» (фарма + электроника)|	Интеграция с внешними компаниями: обмен данными о поставках, использовании оборудования, совместные продукты|	Контракт, поставка, устройство|	Клиники (заказ препаратов, оборудования), финтех (финансирование поставок)|
|Домен «Аналитика и отчёты» (витрина самообслуживания)|	Предоставление агрегированных, безопасных данных для BI и портала самообслуживания. Медицинские карты не включаются	|Агрегированные витрины (выручка по клиникам, ROI финтех-продуктов, загрузка ИИ)|	Головной офис, руководители доменов, аналитики|
|Домен «Инфраструктура и интеграция»|	Облачная шина данных (Kafka), API Gateway, каталог метаданных, безопасность|	События, потоки, роли доступа|	Все домены|

Почему именно такое разделение?

	Оно соответствует организационной структуре и бизнес-направлениям (клиники, финтех, ИИ — отдельные юрлица/подразделения).

	Позволяет развивать каждый домен независимо: финтех может менять кредитные продукты, не затрагивая клиники; ИИ-команда переобучает модели, не трогая DWH.

	Соблюдает регуляторные требования — медицинские карты остаются строго внутри домена «Клиники» и не попадают в общую аналитическую витрину.

	Готовит почву для будущих партнёров — домен «Партнёры» изолирует специфику внешних интеграций.

4. Обоснование разделения на домены и преимущества для бизнеса

4.1. Логика разделения
	Бизнес-способности (Business Capabilities): Каждый домен соответствует отдельной бизнес-способности компании: «Лечение пациентов», «Финансовые услуги», «Искусственный интеллект», «Работа с партнёрами». Это классический подход Domain-Driven Design (DDD).

	Изменяемость: Частота изменений в клиниках (регулярные обновления протоколов) отличается от финтеха (сезонные акции) и ИИ (частый переобучение моделей). Отдельные домены позволяют менять каждый сервис в своём ритме.

	Командная автономия: В компании уже существуют отдельные подразделения (клиники, финтех, ИИ-компания). Архитектура должна отражать организационную структуру (закон Конвея).

	Безопасность и регуляторика: Медицинские данные требуют особой защиты (ФЗ-323, HIPAA). Смешивание их с финансовыми в одном DWH создаёт риски. Доменная изоляция — единственный надёжный способ.

4.2. Преимущества для бизнеса

|Бизнес-преимущество|	Как достигается|
|------------------|--------------|
|Ускорение вывода продуктов (time-to-market)|	Команда финтеха может самостоятельно развернуть новый микросервис (например, BNPL-кредиты) и через Kafka начать публиковать события. Старый DWH не является bottleneck'ом.|
|Снижение зависимости от центрального DWH|	DWH теперь используется только для аналитической витрины, а не для операционной бизнес-логики. Тысячи хранимых процедур SQL Server 2008 постепенно выносятся в доменные сервисы.|
|Возможность независимого развития направлений|	ИИ-сервисы могут использовать новейшие фреймворки (PyTorch, TensorFlow) без оглядки на поддержку SQL Server. Клиники переходят с PowerBuilder на современный стек.|
|Масштабирование новых бизнесов|	Чтобы купить фармацевтическую компанию, достаточно выделить для неё отдельный домен и интегрировать через шину. Это займёт недели, а не месяцы.|
|Снижение рисков|	Отказ ESB (Apache Camel) больше не парализует всю компанию — Kafka обеспечивает отказоустойчивость и репликацию. Авария в одном домене не влияет на другие.|
|Улучшение качества данных|	Каждый домен владеет своими «источниками правды». Каталог метаданных (DataHub) фиксирует происхождение данных (lineage), что повышает доверие к отчётам.|

4.3. Переход от текущего состояния к целевому
	Фаза 1 (первые 2 месяца):

		Развернуть Kafka и API Gateway в облаке.

		Выделить домен «Аналитика»: перенести отчётообразование в Snowflake/BigQuery.

		Начать реплицировать данные из старого DWH в новый Data Lake (CDC-инструментами).				

	Фаза 2 (3–6 месяцев):

		Окончательно разделить домены «Клиники», «Финтех», «ИИ» как отдельные микросервисные приложения.

		Перевести операционные интерфейсы с PowerBuilder на новые API.

	Фаза 3 (через год):

		Отказаться от легаси-ESB и SQL Server 2008.

		Подключить первого внешнего партнёра через домен «Партнёры».

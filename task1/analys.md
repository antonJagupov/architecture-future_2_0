1. Поддержка ключевых бизнес-сценариев:

|Сценарий|	Как поддерживается целевой архитектурой|
|---|---|
|Быстрая подготовка отчётности|	Self-service портал поверх современного MPP-хранилища (Snowflake/BigQuery) с семантическим слоем. Отчёты строятся за секунды/минуты.|
|Независимое развитие финтех- и ИИ-направлений|	Чёткие доменные границы, отдельные микросервисы, событийная шина. Изменения в одном домене не ломают другие.|
|Масштабирование для новых бизнесов|	Облачная инфраструктура, Data Lake, доменная архитектура. Новый бизнес легко подключить как новый домен.|
|Соблюдение требований регуляторов|	Каталог данных управляет правами доступа на уровне записей. Медицинские данные изолированы в домене «Клиники» и не попадают в аналитическую витрину.|
|Отказ от легаси|	Постепенная миграция: PowerBuilder и старый DWH замещаются новыми сервисами, ESB заменяется на Kafka.|

2. Проблемные места текущей архитектуры и их влияние на бизнес-сценарии
|№|	Проблема|	Описание|	Пострадавшие бизнес-сценарии|
|----|----|----|----|
|1|	Медленное формирование отчётов|	DWH на SQL Server 2008 не справляется с объёмом (сотни ТБ). Сложные отчёты строятся часами.	|Управленческие решения задерживаются; аналитики тратят время на ожидание вместо анализа.|
|2|	Бизнес-логика в DWH|	Тысячи хранимых процедур и трансформаций. Изменение отчёта требует переписывания ETL, риски поломки.	|Time-to-market новых фич – недели/месяцы. Быстрая реакция на требования рынка невозможна.|
|3|	Монолитность ESB|	Apache Camel – единая точка интеграции. Любое изменение требует координации между всеми командами.	|Замедление интеграции новых сервисов (например, покупка фарм-компании).|
|4|	Устаревшие технологии|	SQL Server 2008 (без поддержки), PowerBuilder (нет специалистов), низкая производительность.	|Риски ИБ, высокая стоимость поддержки, невозможность масштабирования.|
|5|	Отсутствие self-service|	Все отчёты готовят ИТ-специалисты. Бизнес-пользователи не могут построить произвольный срез.	|Низкая автономность бизнес-подразделений, ИТ-департамент – узкое место.|
|6|	Перемешивание данных|	Медицинские, финансовые и инвентаризационные данные в одном DWH без чётких доменов.	|Конфликты при параллельной разработке; сложно разграничить доступ к чувствительным данным.|
|7|	Отсутствие стандартизации|	Нет единого каталога данных, линеджа, глоссария. Трудно понять, откуда взялась цифра в отчёте.	|Ошибки в отчётности, низкое доверие к данным.|

3. Приоритизация проблем (матрица Эйзенхауэра + MoSCoW)
Используем матрицу Эйзенхауэра по осям «Срочность» и «Важность» (важность = влияние на выручку, скорость выхода на рынок, качество сервисов). Затем группируем в MoSCoW.

|Проблема|	Срочность|	Важность|	Квадрант|	Приоритет MoSCoW|	Обоснование|
|--------|---------|-----------|----------|------------|-----------|
|1. Медленные отчёты|	Высокая|	Высокая|	Сделать сейчас|	Must have|	Руководство не может быстро оценить эффективность финтеха и клиник → потери выручки.|
|3. Монолитность ESB|	Высокая|	Высокая|	Сделать сейчас|	Must have|	Блокирует интеграцию новых бизнесов (покупка фарм-компаний). Без событийной шины нельзя масштабироваться.|
|4. Устаревшие технологии|	Средняя|	Высокая|	Спланировать|	Should have|	Риск безопасности и нехватка специалистов, но можно временно митигировать за счёт облачной миграции.|
|2. Бизнес-логика в DWH|	Средняя|	Высокая|	Спланировать|	Should have|	Замедляет time-to-market, но возможно постепенное вынесение бизнес-логики в микросервисы.|
|5. Отсутствие self-service|	Низкая|	Высокая|	Спланировать|	Should have|	Важно для автономности бизнеса, но без быстрых отчётов и чистых данных бессмысленно.|
|6. Перемешивание данных|	Низкая|	Средняя|	Делегировать|	Could have|	Решается после внедрения доменов, пока можно жить с дополнительными процедурами согласования.|
|7. Отсутствие стандартизации|	Низкая|	Средняя|	Делегировать|	Could have|	Улучшает качество, но не критично для MVP.|

Итоговая приоритизация (MoSCoW):

	Must have (критично для запуска через 2 месяца):

		Проблема 1: Медленные отчёты → решение: замена DWH на современное MPP-хранилище в облаке.

		Проблема 3: Монолитность ESB → решение: внедрение событийной шины (Kafka) и API Gateway.

	Should have (важно, но можно отложить на 3–6 месяцев):

		Проблема 4: Устаревшие технологии → миграция с SQL Server 2008 и PowerBuilder.

		Проблема 2: Бизнес-логика в DWH → рефакторинг, вынос в доменные сервисы.

		Проблема 5: Self-service портал → разработка витрины с семантическим слоем.

	Could have (хорошо бы, но не критично):

		Проблема 6: Перемешивание данных → чёткое выделение доменов в аналитике.

		Проблема 7: Каталог данных – внедрять после стабилизации архитектуры.

Связь с целями бизнеса:

	Выручка: Must have – быстрые отчёты позволяют руководству оперативно корректировать стратегию финтеха и клиник; новая шина данных даёт возможность подключать платные ИИ-сервисы для внешних клиентов.

	Скорость выхода на рынок: Should have – вынос бизнес-логики из DWH и переход на микросервисы сокращают time-to-market новых продуктов с месяцев до недель.

	Качество сервисов: Must have – современная шина данных и облачная инфраструктура повышают надёжность и отказоустойчивость критичных систем (финтех-транзакции, ИИ-диагностика).

Вывод: в первую очередь команда должна сосредоточиться на замене ESB на событийную шину и миграции аналитической нагрузки в облачное MPP-хранилище (например, Snowflake или BigQuery). Это позволит уже через 2–3 месяца кардинально ускорить отчётность и обеспечить интеграционную готовность для новых бизнесов. Остальные задачи планируются на следующие кварталы.
